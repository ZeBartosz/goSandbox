Session 001 · map before mastery

Go concurrency primitives are a small toolbox, not magic.

Your interview target for now: when you see work happening at the same time, choose the primitive that expresses the coordination clearly. This first explainer gives you the map; the exercises give you the muscle memory.

The one sentence model

A goroutine runs a function concurrently; a channel lets goroutines communicate; a WaitGroup waits for goroutines to finish; a Mutex protects shared data; an atomic updates one small value with low-level synchronization.

Go's official material introduces goroutines and channels in the Go Tour concurrency section, documents locks and wait groups in package sync, and documents atomics in package sync/atomic.

Primitive map

Goroutine

go f() starts f concurrently. It does not automatically give you a result, wait, or handle errors.

Channel

A typed pipe. Send with ch <- x; receive with x := <-ch. Channels are useful when ownership or data should move between goroutines.

WaitGroup

A counter for "how many goroutines are still running?" Add before starting, call Done when finished, Wait in the parent.

Mutex

A lock around shared memory. If multiple goroutines read/write the same map, slice, struct, or counter, a Mutex is often the straightforward fix.

Atomic

A specialized synchronized operation for one value, commonly counters or flags. Prefer it when the state is truly tiny and independent.

Try this mentally

You are adding parallel lexing to a toy compiler: 4 goroutines tokenize 4 files. You need to gather each file's token count. Which primitive is essential?

A goroutine for each file, plus either a channel to return token counts or a shared result slice protected by a Mutex. A WaitGroup is needed if the parent must wait for all work to finish. The clean beginner solution: channel of results + WaitGroup/close.
