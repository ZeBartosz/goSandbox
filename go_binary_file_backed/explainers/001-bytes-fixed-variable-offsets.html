LunaSQL storage step 3

Rows are just agreements about bytes

Your current LunaSQL table is JSON: self-describing, easy to inspect, and expensive. A binary row store removes most labels and stores values at agreed byte positions. This lesson is the bridge from []map[string]string to rows that live directly inside a .table file.

1. A byte is the smallest addressable storage unit you will handle

In Go, a []byte is a slice over bytes; the official Go slices article explains that a slice is a descriptor containing a pointer, length, and capacity, not the backing array itself [Go slices internals].

If a file contains these bytes:

4c756e61

Those bytes can mean the string "Luna" only because we choose to decode them as UTF-8 text. Bytes do not carry meaning by themselves.

2. Fixed-size fields make offsets easy

A fixed-size field always uses the same number of bytes. A uint32 is 4 bytes. That means if a row layout is:

FieldSizeOffset
id4 bytes0
age4 bytes4

Then row {id: 7, age: 42} is always 8 bytes long. Go's encoding/binary package is the standard tool for putting fixed-size numbers into byte slices [Go encoding/binary].

row := make([]byte, 8)
binary.LittleEndian.PutUint32(row[0:4], 7)
binary.LittleEndian.PutUint32(row[4:8], 42)

id := binary.LittleEndian.Uint32(row[0:4])
age := binary.LittleEndian.Uint32(row[4:8])

Little endian means the least significant byte comes first. The exact choice matters less than being consistent.

3. Variable-size fields break simple offsets

A string is variable-size: "Al" is 2 bytes, "Bartosz" is 7 bytes. If a row stores id followed directly by name, where does the next field start? It depends on the name length.

The usual beginner solution is length-prefixing:

PartSizeMeaning
id4 bytesfixed integer
name_len4 bytesnumber of bytes in name
name_bytesname_len bytesraw string bytes

Now the decoder reads 4 bytes for id, 4 bytes for name_len, then exactly name_len bytes for the name.

4. Offsets are positions, not values

An offset is a byte position measured from a known start. If a row starts at byte 100 in a file, then offset 4 inside the row means absolute file position 104.

absolutePosition = rowStart + fieldOffset

SQLite's file format documentation is full of precise byte offsets because a database file must be understandable without guessing [SQLite file format]. PostgreSQL documents pages similarly, using page headers and item pointers to find tuples [PostgreSQL page layout].

5. A tiny LunaSQL row format

For the next exercise, use this append-only row format:

row_size uint32
id       uint32
name_len uint32
name     []byte

The outer row_size lets the scanner skip or read one whole row at a time. The inner name_len lets the decoder know where the variable-size string ends.

This is not a final database design. It is deliberately pre-page, pre-index, and pre-transaction so you can master bytes first.
